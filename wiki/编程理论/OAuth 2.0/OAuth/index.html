授权登录

背景

随着国美生态系统中APP的不断增加，多个APP的登录问题日渐突出，比如我们新做一个美店APP，只有两个选择

  1. 重新开放美店的登录系统
  2. 复用现有国美APP的代码

对于方案1，不仅需要增加开发成本，并且不一定能有效的接入国美会员体系。
对于方案2，也就是我们现在使用的方案，国美APP提供了登录SDK，美店APP集成，这就存在严重的不足，首先是APP的包会变大，其次每次有新的APP需要集成登录SDK，都需要前端后端开发一起开发，以便区分什么APP集成，做出对应的兼容措施，再次是SDK变化，集成SDK的APP不能及时集成，最后，无法保证其他APP能否安全的保护用户的信息。

所以我们就考虑，有没有更有效的方案呢？

参考微信、QQ三方授权，发现，通过OAuth授权方案可以解决上面的登录问题，同时该方案保证了安全性。

OAuth 简介

OAuth 2.0 是目前最流行的授权机制，用来授权第三方应用，获取用户数据。该机制用到了很多术语，为了方便，我将术语列举在下面

  1. Third-party application：第三方应用程序，本文中又称”客户端”（client），可以想象为美店APP。
  2. HTTP service：HTTP服务提供商，可以假象为国美后端服务。
  3. Resource Owner：资源所有者，可以假想为过么的用户。
  4. Authorization server：认证服务器，即服务提供商专门用来处理认证的服务器。
  5. Resource server：资源服务器，即服务提供商存放用户生成的资源的服务器。

我将上面5个角色画了一张图，方便理解：

简单说，OAuth 就是一种授权机制。数据的所有者告诉系统，同意授权第三方应用进入系统，获取这些数据。系统从而产生一个短期的进入令牌（token），用来代替密码，供第三方应用使用。

后端设计

使用code换token的请求，对应于【总体设计图中】中的步骤H、I，包含以下参数：

  1. grant_type：表示使用的授权模式，必选项，此处的值固定为”authorization_code”。
  2. code：表示上一步获得的授权码，必选项。
  3. redirect_uri：表示重定向URI，必选项，且必须与A步骤中的该参数值保持一致。
  4. client_id：表示客户端ID，必选项。

使用code换token的响应，对应于【总体设计图中】中的步骤J，包含以下参数：

  1. access_token：表示访问令牌，必选项。
  2. token_type：表示令牌类型，该值大小写不敏感，必选项，可以是bearer类型或mac类型。
  3. expires_in：表示过期时间，单位为秒。如果省略该参数，必须其他方式设置过期时间。
  4. refresh_token：表示更新令牌，用来获取下一次的访问令牌，可选项。
  5. scope：表示权限范围，如果与客户端申请的范围一致，此项可省略。
